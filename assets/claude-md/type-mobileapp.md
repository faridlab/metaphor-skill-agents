# Metaphor Mobile App

> Type: **`mobileapp`** — Kotlin Multiplatform (KMP) app targeting Android + iOS. Compose Multiplatform UI, Ktor networking, SQLDelight offline-first, Koin DI.
> This file orients Claude. Skills carry depth; load them on demand.

## What this is

A KMP app with shared business logic in `commonMain`, platform code in `androidMain` / `iosMain`. UI is Compose Multiplatform + Material 3. Networking is Ktor (OkHttp on Android, Darwin on iOS). Storage is SQLDelight. DI is Koin. Offline-first with a sync engine that reconciles local state with the backend service.

## Golden path

```bash
# Android
./gradlew :android:installDebug

# iOS (from Xcode; or pre-build shared binaries)
./gradlew :shared:iosSimulatorArm64Binaries

# Tests
./gradlew :shared:commonTest
./gradlew :android:testDebugUnitTest

# Lint
./gradlew detekt
```

Inside a Metaphor workspace, `metaphor dev serve` / `metaphor dev test` may wrap these.

## Rules

- **MUST** keep domain / application code in `commonMain`. Platform-specific code only in `androidMain` / `iosMain` (biometrics, keychain, file paths, push tokens).
- **MUST** use `expect` / `actual` for any API that differs per platform (crypto, secure storage, biometric auth).
- **MUST** use Koin modules — one per layer (data, domain, presentation). Register in `DiModule.kt`.
- **MUST** use SQLDelight for local persistence. Never hand-roll SQL strings; always `.sq` files with typed queries.
- **MUST** read API base URL / keys from `local.properties` (gitignored), not hardcoded. Emulator defaults: Android `http://10.0.2.2:3000`, iOS `http://localhost:3000`.
- **MUST** keep domain layer free of framework imports (no Compose, no Ktor, no SQLDelight) — domain is pure Kotlin.
- **MUST** mark custom (non-regenerated) zones with `// <<< CUSTOM` / `// END CUSTOM` when the file is regenerated by `backbone-mobilegen` or `metaphor make`.
- **SHOULD** follow MVI pattern with a `BaseViewModel` per screen; state flows are `StateFlow<UiState>`.
- **NEVER** make blocking calls on the main thread — use `viewModelScope` + suspend functions.
- **NEVER** bypass the sync engine for writes that need server reconciliation — queue them, don't fire-and-forget.
- **NEVER** commit `local.properties`, Google service files, signing configs, or API keys.

## Tech stack (non-negotiable)

- **Kotlin Multiplatform** (Gradle 8.13+); Kotlin 2.x.
- **Compose Multiplatform** + **Material 3**.
- **Ktor client** — OkHttp engine on Android, Darwin engine on iOS.
- **SQLDelight 2.x** for persistence; migrations in `.sq` files.
- **Koin 4.x** for DI.
- **Decompose 3.x** for navigation.
- **Android**: minSdk 24, targetSdk 35; WorkManager for background jobs.
- **iOS**: Xcode 15+; keychain for secrets.

## Common tasks

- "Add a new screen" → create `presentation/screens/<feature>/` with `<Feature>Screen.kt` + `<Feature>ViewModel.kt`; register VM in Koin module; add nav entry in Decompose root.
- "Add a new entity with offline sync" → define SQLDelight `.sq`, create DAO + repository in `infrastructure/`, write domain interface in `domain/`, register sync handler via `SyncEntityRegistry`.
- "Add a platform-specific capability" → define `expect` in `commonMain`, implement `actual` in `androidMain` + `iosMain`.
- "Override API base URL locally" → edit `local.properties` (`API_BASE_URL=http://10.0.2.2:3000`).
- "Run on device" → `./gradlew :android:installDebug` or open Xcode and run the `iosApp` scheme.

## Anti-patterns

- Putting Compose / Ktor / SQLDelight in `domain/` (breaks multiplatform purity).
- Hardcoding API URLs or keys (use `local.properties`).
- Skipping the sync engine and writing ad-hoc networking in ViewModels.
- Modifying generated files outside CUSTOM markers (silently overwritten on regen).
- Blocking calls on the main thread (`runBlocking`, `.get()` on futures).
- Committing `local.properties`, `google-services.json`, or signing keystores.
